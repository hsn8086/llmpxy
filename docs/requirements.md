# 用户系统与订阅管理需求文档

## 1. 背景

当前 `llmpxy` 已具备以下基础能力：

- 多协议入站：
  - `oairesp`
  - `oaichat`
  - `anthropic`
- 多协议出站：
  - `oairesp`
  - `oaichat`
  - `anthropic`
- provider / provider group 路由
- `fallback` / `load_balance`
- 会话存储
- 日志
- HTTP 代理支持

在此基础上，需要新增一套面向业务运营和终端用户自助使用的管理系统，用于支持：

- 用户与用户组
- 多 API Key
- 多有效订阅
- 订阅级余额
- 订阅级限额
- 调用日志
- 用户组与订阅的代理组限制
- 用户前端
- 管理后台

## 2. 总体目标

新增一套完整的业务控制层，位于现有“协议适配 / 请求路由 / provider 调用”之上，实现：

- 对调用方身份进行识别
- 对用户、用户组、API Key、订阅进行权限和范围控制
- 在请求发往上游前进行限额和余额校验
- 在请求完成后记录调用日志并更新余额与统计
- 提供用户端和管理端前端页面

## 2.1 配置边界原则

以下内容属于高频运营数据，不应继续写入静态 `config` 文件，而应改为数据库与管理后台驱动：

- 用户与用户组
- API Key
- 订阅模板与用户订阅实例
- 余额
- 限额
- 价格配置
- `provider_group` 价格倍率
- 运营态权限与范围配置
- provider 与 provider_group 业务配置

静态 `config` 仍适合承载低频基础设施配置，例如：

- 服务监听地址
- 网络代理
- 协议适配基础行为

因此，后续实现需要将现有部分配置逻辑从 `config.py`/TOML 迁移为数据库读取或管理后台维护。

## 3. 系统范围

第一版范围包含：

- 用户账号登录
- 管理员角色
- 用户组
- 用户管理
- 多 API Key 管理
- 多订阅并存
- 订阅级余额
- 订阅级限额
- 代理组限制
- 调用日志
- 用户前端
- 管理后台
- 管理员代充值/代运营

第一版暂不包含：

- 用户自助购买
- 用户自助充值
- 支付系统接入
- 企业组织/多租户体系
- 订阅按 model/provider/protocol 细粒度限制
- API Key 按请求次数限额
- 单次调用 token 上限

## 4. 已确认需求

### 4.1 代理调用认证

代理调用请求支持以下认证方式：

- `Authorization: Bearer <api_key>`
- `X-API-Key`

### 4.2 用户前端登录

用户前端采用：

- 用户名或邮箱 + 密码登录
- JWT 鉴权
- `Access Token + Refresh Token`

### 4.3 后台登录

后台与普通用户共用同一账号体系：

- 用户表统一管理
- 使用角色区分权限
- 至少包含：
  - `user`
  - `admin`

### 4.4 前端项目形态

前端采用：

- 单前端项目
- 前后端分离
- 技术栈：
  - `Vue + Vite`

建议逻辑区域：

- 用户区 `/app/*`
- 管理区 `/admin/*`

### 4.5 用户与用户组

- 一个用户只属于一个用户组
- 用户组用于表达默认权限边界
- 用户组可配置允许访问的 `provider_group`

### 4.6 API Key

- 一个用户可拥有多把 API Key
- API Key 数量不设固定上限
- 每把 API Key 可配置：
  - 到期时间
  - 每日 token 限额
  - 每月 token 限额
  - 订阅范围
  - `provider_group` 范围

API Key 第一版不要求支持：

- 每日请求数
- 每月请求数
- 单次最大 token

### 4.7 订阅模型

- 订阅采用“模板 + 分配实例”模型
- 先创建订阅模板，再分配给具体用户形成用户订阅实例
- 模板用于复用配置，实例用于承载用户侧余额、状态、时间和排序

- 一个用户可以同时拥有多条有效订阅
- 余额归属在订阅级
- 订阅优先级通过排序表达
- 不强调对外暴露数值优先级语义
- 管理员和用户都可以调整订阅顺序

请求到来时：

- 从满足条件的订阅中
- 按排序顺序
- 选择第一个可用订阅进行承接和扣费

### 4.8 订阅适用范围

第一版订阅适用范围只按 `provider_group` 限制。

第一版暂不要求：

- provider 级限制
- model 级限制
- protocol 级限制

### 4.9 用户组与订阅限制规则

当用户组限制和订阅限制同时存在时：

- 最终可用 `provider_group` 取交集

### 4.10 限额策略

- 限额超出时直接硬拒绝
- 第一版至少包含 token 维度限额
- 限额对象包括：
  - API Key
  - 订阅
- 订阅级限额第一版支持：
  - 每日 token
  - 每月 token

### 4.11 调用日志

调用日志第一版需要展示：

- 成功请求
- 被拒绝请求

被拒绝请求包括但不限于：

- 认证失败
- API Key 过期
- API Key 无权限
- 无可用订阅
- 余额不足
- 超出限额
- 订阅过期
- provider group 不允许

### 4.12 用户前端功能

第一版用户前端需要支持：

- 账户信息
- 我的订阅
- 调用记录
- 限额用量
- API Key 管理
- 可用能力说明

账户页允许用户：

- 修改密码
- 修改基础资料
- 修改用户名

基础资料第一版允许维护：

- 显示名
- 邮箱

邮箱要求：

- 全局唯一

### 4.13 用户前端能力说明页

第一版展示内容：

- 可用 `provider_group`
- 可用模型

第一版暂不要求：

- 示例请求
- SDK 示例代码

### 4.14 管理区功能

第一版管理区需要支持：

- 用户组管理
- 用户管理
- 订阅管理
- API Key 管理
- 调用日志查看
- 管理员代运营

管理员代运营第一版包括：

- 给订阅加余额
- 开通订阅
- 调整订阅限额
- 延长订阅有效期
- 重置用户密码

补充规则：

- 订阅必须先创建，再分配给用户
- 订阅模板与用户订阅实例在后台分开管理

### 4.15 订阅模板

第一版订阅模板需要支持预置以下默认字段：

- 名称与描述
- 默认余额
- 默认限额
- 默认 `provider_group` 范围
- 默认有效期时长

模板分配给用户形成实例时，第一版允许覆盖：

- 余额
- 限额
- `provider_group` 范围
- 有效期

### 4.16 充值与购买

第一版不做：

- 用户自助充值
- 用户自助购买

第一版只支持：

- 管理员代充值/代运营

### 4.17 余额扣费

- 第一版余额扣减采用按模型单价扣费
- 系统需要维护价格配置
- 价格配置优先级为：
  - 先按模型匹配价格
  - 若模型未命中，则按 `provider_group` 回退价格
- `provider_group` 需要支持价格倍率配置
- 价格倍率第一版采用乘法倍率
- 未配置倍率时默认按 `1.0` 处理
- 当命中模型单价时，仍可叠加 `provider_group` 价格倍率
- 当未命中模型单价而回退到 `provider_group` 价格时，倍率同样生效
- 实际扣费依据请求实际 usage 计算
- 价格规则至少需要支持按模型区分
- 第一版余额与价格单位采用积分点数，不直接绑定法币

### 4.17.1 网络代理配置

- 除基础设施级网络代理配置外，业务侧还需要支持更细粒度的代理配置
- 提供商需要支持单独设置网络代理
- 提供商级网络代理配置属于高频运营数据，不应继续写入静态 `config`
- 最终上游请求构造时应能够读取并应用提供商自己的代理设置

第一版 provider 级应支持维护的字段至少包括：

- 代理地址
- 总超时配置
- 启停状态

provider 第一版还需要支持：

- 协议类型可编辑
- 模型映射数据库化管理
- 模型白名单模式
- 单条模型映射可启停

### 4.18 Refresh Token

- 前端登录采用 `Access Token + Refresh Token`
- `Access Token` 第一版有效期为 2 小时
- `Refresh Token` 第一版有效期为 7 天
- 第一版 `Refresh Token` 采用纯 JWT，不落库
- 第一版不要求支持服务端撤销单个 refresh token
- 第一版 `Refresh Token` 采用轮换续签
- 每次刷新成功后应签发新的 refresh token，并使旧 token 失效

### 4.19 用户名规则

- 用户允许修改用户名
- 用户名必须全局唯一
- 用户名仅允许：
  - 字母
  - 数字
  - 下划线
- 用户名长度限制为 3 到 32
- 用户名唯一性校验采用大小写不敏感规则

### 4.20 API Key 订阅范围规则

- API Key 的订阅范围第一版同时支持两种模式：
  - 使用用户当前全部可用订阅
  - 显式选择部分订阅
- 如果 API Key 配置了显式订阅范围，则请求只能在该范围内选择承接订阅

### 4.21 API Key 编辑能力

- 用户前端第一版允许在创建 API Key 后继续编辑其 scope 和限制项
- 第一版允许编辑的重点内容包括：
  - 订阅范围
  - `provider_group` 范围
  - 到期时间
  - token 限额

API Key 生命周期规则：

- 禁用可恢复
- 删除为永久删除，不可恢复

### 4.22 日志保留策略

- 调用日志第一版默认保留 90 天
- 过期日志应支持定期清理
- 调用日志筛选第一版维持已确认的基础筛选项，不再扩展到订阅和协议维度
- 异步导出任务与导出文件默认保留 7 天
- 异步导出文件第一版存储在本地文件系统
- 异步导出文件仅允许发起者本人和管理员下载

### 4.23 前端权限边界

- 普通用户只能查看和操作自己的数据
- 管理员可以查看全局用户数据
- 普通用户不可访问管理区
- 管理员可以进入用户区查看自己的普通用户页面
- 用户前端第一版不展示 provider_group 的网络代理相关说明

### 4.24 管理员密码重置

- 第一版管理员可以直接为用户设置新密码

### 4.25 订阅排序提交方式

- 第一版订阅排序接口采用“提交完整顺序数组”方式
- 后端负责将顺序重排为连续且稳定的内部排序值

### 4.26 价格配置页面

- 管理区第一版需要单独的价格配置页面
- 价格配置页面至少应支持：
  - 模型价格配置
  - `provider_group` 回退价格配置
  - `provider_group` 价格倍率配置
- 第一版价格配置页至少支持手工新增、编辑与停用价格规则
- 价格规则第一版保存后立即生效
- 被停用的价格规则对后续新请求应立即失效
- 价格规则第一版需要支持显式排序，以控制匹配优先级

### 4.27 用户区扣费展示

- 用户区调用记录第一版需要展示每次调用的积分扣费明细
- 扣费展示应与 token 使用量同时出现，便于用户理解消费来源

用户区订阅列表第一版还需要：

- 展示订阅实例来源模板信息
- 第一版仅展示模板名称

### 4.28 调用日志导出

- 调用日志第一版需要支持异步导出
- 导出能力应适配大批量数据场景，不要求同步一次性返回全部结果

### 4.29 配置迁移原则

- 动态路由、计费、限额、订阅范围、价格与价格倍率等业务规则第一版全部从数据库读取
- 静态 `config` 不再承载上述高频运营规则
- `config` 只保留低频基础设施配置
- `provider` 第一版也应数据库化，不再以静态 `config` 作为业务配置来源
- `provider_group` 第一版也应数据库化，不再以静态 `config` 作为运营配置来源

### 4.29.1 Provider 与 ProviderGroup 管理关系

- 第一版后台中由 `provider_group` 直接维护成员 provider 列表
- provider 与 provider_group 的归属关系以 group 侧配置为准
- provider_group 第一版需要独立启停状态

provider_group 管理页第一版至少需要支持：

- group 列表
- group 新增与编辑
- group 启停
- 成员 provider 维护
- 当前价格倍率展示
- 当前回退价格展示
- 成员 provider 状态汇总展示

停用提示要求：

- 管理区需要展示 provider_group 停用状态与停用原因
- 用户区若某些能力因 group 停用不可用，也需要展示停用原因
- 若某个 provider 被停用且影响用户能力，用户区第一版也需要展示停用原因
- provider_group 停用原因第一版采用简短文本字段，长度上限 200 字

### 4.29.2 Provider 管理原则

- `provider` 第一版整体数据库化管理
- provider 的模型映射也数据库化，不再放在静态 `config`
- provider 的协议类型在后台允许编辑
- 后台需要支持 provider 级模型映射编辑
- 后台需要支持 provider 级模型白名单模式
- provider 级超时配置第一版只要求总超时
- 后台需要支持 provider 测试连接功能

provider 测试连接第一版要求：

- 以拿到 2xx 响应作为成功标准
- 返回测试耗时
- 展示失败错误信息
- 测试时带上当前 provider 的代理配置
- 允许手工指定测试路径

默认建议路径策略：

- 按协议类型提供默认测试路径
- 同时允许管理员手工覆盖测试路径
- 第一版默认覆盖以下协议：
  - `oaichat`
  - `oairesp`
  - `anthropic`

provider 模型映射第一版规则：

- 模型映射不要求显式排序
- 第一版以精确映射为主

### 4.30 邮箱验证

- 第一版邮箱字段不要求验证流程

### 4.31 价格审计快照

- 调用日志第一版需要保存命中的价格与倍率快照
- 价格规则后续发生变化时，不应影响历史调用日志的审计信息

## 5. 业务规则草案

### 5.1 请求处理链路

请求进入系统后的业务处理顺序暂定为：

1. 识别 API Key
2. 校验 API Key 是否存在、有效、未过期、未禁用
3. 找到所属用户
4. 校验用户状态
5. 获取用户组
6. 获取用户当前所有有效订阅
7. 根据 API Key 的订阅范围过滤订阅集合
8. 根据 API Key 的 `provider_group` 范围过滤
9. 根据用户组可用 `provider_group` 过滤
10. 根据订阅可用 `provider_group` 过滤
11. 对剩余可用订阅按排序顺序选择第一个
12. 检查 API Key token 限额
13. 检查订阅限额
14. 检查订阅余额
15. 计算本次请求可使用的 route override
16. 调用现有 `dispatcher`
17. 记录调用日志
18. 扣减订阅余额
19. 更新 API Key / 订阅的统计与限额计数

### 5.2 动态路由

现有项目使用全局 route。

后续需要新增请求级动态 route override，来源于：

- 用户组限制
- 订阅限制
- API Key 订阅范围
- 当前订阅排序结果

第一版动态路由相关业务规则应从数据库读取，不再依赖静态 `config` 维护运营态配置。

动态路由阶段还需要综合考虑提供商级网络代理配置，以生成最终上游请求参数。

若 provider 被停用，则其不再参与后续新请求路由；已在执行中的请求第一版不做强制中断。
若 provider_group 被停用，则其不再参与后续新请求路由；已在执行中的请求第一版不做强制中断。

provider 模型映射在匹配时还需要考虑单条映射启停状态。

### 5.3 订阅选择规则

第一版规则为：

- 订阅模板定义默认配置
- 用户订阅实例承载实际生效数据
- 多条订阅可同时生效
- 从满足条件的订阅中按排序顺序选择
- 不区分定向优先还是通用优先
- 不根据模型、provider 或 protocol 进行订阅匹配

### 5.4 扣费规则

- 请求完成后根据实际 usage 计算扣费金额
- 扣费金额的价格匹配顺序为：
  - 优先按模型价格
  - 未命中时按 `provider_group` 价格回退
- `provider_group` 可配置价格倍率
- 价格倍率采用乘法方式生效
- 未配置倍率时默认按 `1.0` 处理
- 最终扣费金额需要将基础价格与 `provider_group` 倍率共同纳入计算
- 实际扣费从本次选中的承接订阅余额中扣减
- 如果余额不足，则请求在发往上游前应被拒绝
- 调用日志需要保存当次命中的价格、倍率与最终扣费快照
- 被停用的价格规则不再参与后续新请求匹配

价格规则优先级第一版定义为：

- 先按规则层级匹配：模型价格优先于 `provider_group` 回退价格
- 同一层级内再按显式排序决定命中顺序
- 同一规则层级内顺序值必须唯一
- 若后台保存时发现同层级排序冲突，应直接阻止保存

### 5.5 API Key 订阅过滤

- API Key 若配置为“全部订阅”，则可参与用户全部当前可用订阅的筛选
- API Key 若配置为“部分订阅”，则只允许从绑定订阅中选择承接订阅

### 5.6 API Key provider_group 过滤

- API Key 第一版还支持 `provider_group` 范围限制
- 若 API Key 配置了允许的 `provider_group` 集合，则请求只能在该集合内路由
- API Key 的 `provider_group` 限制与用户组、订阅限制共同生效

### 5.7 API Key 生命周期

- API Key 支持禁用与删除两种操作
- 禁用后的 API Key 不可用于请求，但可恢复启用
- 删除后的 API Key 不可恢复，且不再允许用于认证

### 5.8 Refresh Token 轮换

- 第一版 refresh token 采用轮换续签策略
- 客户端使用 refresh token 换取新 access token 时，应同时获得新 refresh token
- 旧 refresh token 在刷新后应立即失效
- `Access Token` 有效期为 2 小时
- `Refresh Token` 有效期为 7 天

### 5.9 订阅排序更新

- 订阅排序更新接口接收完整顺序数组
- 后端以完整顺序为准进行重排
- 重排后应保证内部顺序值连续、稳定且无重复

## 6. 页面规划草案

### 6.1 用户区页面

建议包含：

- `/login`
- `/app`
- `/app/account`
- `/app/subscriptions`
- `/app/request-logs`
- `/app/usage`
- `/app/api-keys`
- `/app/capabilities`

- 总余额与订阅余额摘要
- 今日 token 消耗
- 本月 token 消耗

建议包含：

- `/admin`
- `/admin/users`
- `/admin/user-groups`
- `/admin/providers`
- `/admin/subscription-templates`
- `/admin/user-subscriptions`
- `/admin/pricing`
- `/admin/api-keys`
- `/admin/request-logs`

其中 provider 管理页第一版至少需要支持：

- provider 列表
- provider 新增/编辑
- provider 启停
- 测试连接
- 模型映射列表
- 模型映射新增/编辑
- 模型映射搜索
- 白名单模式状态展示

provider_group 管理页第一版至少需要支持：

- group 列表
- group 新增/编辑
- group 启停
- 成员 provider 维护
- 价格倍率展示
- 回退价格展示
- 成员 provider 状态汇总展示
